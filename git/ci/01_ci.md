# Integração Contínua (CI)

Pensa assim: antes da cultura DevOps, cada dev fazia suas mudanças isolado, empurrava tudo no fim do sprint e esperava “dar certo”. E claro, nunca dava certo. O código não encaixava, testava tudo de uma vez, quebrava a aplicação inteira… caos.

A **Integração Contínua** nasceu justamente pra evitar esse colapso.

Ela é um **processo automatizado** que garante que *toda vez que alguém integra código novo ao repositório*, o projeto é imediatamente:

* validado
* testado
* analisado
* e construído de forma limpa

Em outras palavras: cada mudança é tratada como algo sério e verificado na hora. Nada de jogar código no repositório e rezar.

---

### O coração da CI: automatizar a dor

No fluxo DevOps, CI significa:

1. **Alguém fez um push ou abriu uma PR.**
2. **Um pipeline automático dispara.**
3. Esse pipeline roda:

   * build do projeto
   * testes unitários
   * testes de integração (dependendo do time)
   * análise estática (SonarQube, por exemplo)
   * checagem de estilo
4. Se tudo passar: beleza, o time pode integrar.
5. Se algo falhar: a PR trava, e a pessoa que enviou precisa arrumar.

CI é praticamente o “porteiro bravo” do repositório.
Ninguém entra se não estiver arrumado.

---

### Mas qual é o papel disso no DevOps?

No DevOps, a ideia é **não ter muro entre dev e ops**.
E CI é uma das peças que elimina esse muro.

Por quê?

* Garante que o código que chega na main *já é confiável*
* Reduz o risco de deploy quebrado lá na frente
* Torna o desenvolvimento muito mais rápido, porque o time pega erro na hora, não um mês depois
* Facilita automação de deploy (que vira CD: Continuous Delivery ou Deployment)

Sem CI, não existe fluxo DevOps de verdade.
É aquele tipo de coisa que parece chata, mas vira a espinha dorsal da equipe.

---

### Um exemplo prático

Imagine um projeto Java com Maven e testes JUnit.

Você cria uma branch → abre PR → e o GitHub dispara um pipeline (Actions) com:

* `mvn clean install`
* rodar JUnit
* verificar cobertura mínima
* lint ou formatador
* talvez análise do Sonar

Se passar, a PR ganha aquele check “verde”, indicando que a mudança tá saudável.

Se não passar, vem aquele “X vermelho” que te proíbe de mergear até arrumar.

Isso é CI na prática: **um guardião automático da qualidade**.

---

CI não é só ferramenta.
É um *compromisso cultural* de que:

* você integra o código cedo
* integra sempre
* integra com qualidade
* deixa máquinas cuidarem da validação chata
* e mantém o projeto sempre num estado “pronto para deploy”

Essa mentalidade muda completamente como equipes trabalham.
