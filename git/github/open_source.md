# Contribuições em Projetos Open Source

Contribuir para projetos open source é quase como entrar numa cidade que já existe: tem regras, tem trânsito, tem bairros diferentes… e você chega com vontade de construir uma nova praça ou consertar um semáforo quebrado.
Issues e PRs são justamente o **idioma universal** dessa comunidade — é assim que você se comunica, pede permissão, debate ideias e entrega melhorias.

## O papel das **issues** no open source

Issues são o **ponto de encontro**. Elas servem para:

* **reportar bugs**
* **sugerir novas features**
* **debater design de solução**
* **pedir esclarecimento**
* **planejar tarefas**
* **levantar dúvidas técnicas**

Por que isso é importantíssimo?

1. **Evita trabalho duplicado**
   Antes de começar a codar, você abre (ou lê) uma issue.
   Assim, você não perde dias implementando uma ideia que já estava sendo feita por outra pessoa.

2. **Mantém alinhamento com os maintainers**
   Abrir uma issue é seu jeito de dizer:
   “Ei, quero mexer nisso — tudo bem pra vocês?”

3. **Dá transparência pra comunidade**
   Todo mundo acompanha, comenta, sugere melhorias.

4. **Garante que sua PR será bem-vinda**
   Contribuir sem alinhamento prévio costuma resultar em PR recusada.

Ou seja:
**Open source não é só escrever código; é conversar antes.**

## O papel dos **pull requests (PRs)**

Se a issue é a conversa inicial, o PR é a entrega.

Ele serve para:

* apresentar sua solução,
* justificar o que você alterou,
* mostrar commit por commit,
* pedir revisão,
* receber feedback,
* ser aprovado (ou ter ajustes solicitados),
* finalmente ser integrado.

PRs bons são como pequenas histórias: claros, objetivos e contextualizados.

### Por que PRs são tão importantes no open source?

1. **Código revisado = código melhor**
   Manter um padrão de qualidade é crucial em projetos grandes.

2. **Reduz risco**
   Uma PR ruim pode introduzir bugs gigantes. Revisão é o filtro.

3. **Ensina e forma pessoas**
   Revisões de PRs são quase mini-aulas.
   Você aprende muito lendo e recebendo feedback.

4. **Registra sua contribuição**
   Cada PR aceita é um item público no seu histórico profissional.

## Como funciona o fluxo profissional típico no open source

1. **Você encontra um problema ou ideia**
   Primeiro passo: procurar no repositório se já existe uma issue.

2. **Se não existir, cria uma issue**
   Explica contexto, motivação, benefício e o que pretende fazer.

3. **Discussão com maintainers**
   Eles validam, sugerem alternativa, redefinem escopo, etc.

4. **“Assigned to you”**
   Eles podem te designar oficialmente para a tarefa.

5. **Você cria uma branch com base na issue**
   Ex: `feat/145-improve-auth-check`

6. **Commits simples, claros, mencionando a issue**
   `fix: adjust token expiration logic (Fixes #145)`

7. **Abre a PR**
   Relaciona a issue → a magia acontece (ao merge, ela fecha sozinha).

8. **Revisão**
   Você recebe feedback, implementa ajustes e conversa com o time.

9. **Merge**
   E pronto: contribuição registrada, issue fechada.

Isso cria disciplina, organização e confiabilidade.
